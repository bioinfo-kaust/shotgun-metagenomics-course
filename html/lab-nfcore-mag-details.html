nf-core/mag: Technical Deep-Dive

Explore the workflow internals, navigate output folders and the Nextflow work directory, read log files, troubleshoot common failures, interpret execution reports, and customise resource limits and tool parameters using config files.

Learning Objectives

  • Understand how Nextflow executes nf-core/mag processes on Ibex
  • Navigate the output directory tree and the work/ directory
  • Read .nextflow.log and per-process log files to diagnose failures
  • Troubleshoot common pipeline errors: OOM kills, missing databases, container failures
  • Generate and interpret HTML execution reports, trace files, and DAG diagrams
  • Override process resources (CPUs, memory, walltime) with a custom config
  • Pass tool-specific arguments to individual pipeline tools via ext.args

Workflow Internals

When you run nextflow run nf-core/mag, Nextflow reads the pipeline DSL2 code and builds a directed acyclic graph (DAG) of processes connected by channels. Each process is a self-contained shell script that is submitted as a separate SLURM job on Ibex. Nextflow tracks dependencies: a binning process only starts once its upstream assembly process has finished and written its output files.

ConceptWhat it means in practice
ProcessA single tool invocation (e.g. MEGAHIT, MetaBAT2). Gets its own SLURM job, its own work/ subdirectory, and its own log files.
ChannelA data stream that connects process outputs to process inputs. Nextflow monitors channels and launches downstream processes as soon as their input is ready.
Work directoryNextflow creates a unique hash directory under work/ for every process execution. All inputs are symlinked there; all outputs are written there.
CacheOn -resume, Nextflow compares inputs, parameters, and container digests. Unchanged processes are skipped and their cached outputs are reused.
ProfileA named set of configuration directives. -profile kaust sets the executor to SLURM, enables Singularity, and points to Ibex scratch storage.

Running the Workflow

The pipeline is always launched from a login node as a SLURM batch script. Nextflow itself uses only a few CPUs and a small amount of memory; the heavy compute happens in the child SLURM jobs it submits. The minimum recommended SLURM allocation for the Nextflow driver process is 4 CPUs / 8 GB RAM.

cat > run_nfcore_mag.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=nfcore_mag
#SBATCH --partition=batch
#SBATCH --time=72:00:00
#SBATCH --mem=8G
#SBATCH --cpus-per-task=4
#SBATCH --output=nfcore_mag_%j.log
#SBATCH --error=nfcore_mag_%j.err

module load nextflow singularity

nextflow run nf-core/mag \
    -r 5.4.2 \
    -profile kaust \
    -resume \
    --input samplesheet.csv \
    --outdir results/ \
    --host_fasta /ibex/reference/KSL/human/GRCh38/GRCh38.fa \
    --skip_spades false \
    --run_virus_identification true \
    --genomad_db /ibex/reference/KSL/genomad/v1.5/genomad_db \
    --skip_gunc false \
    --checkm2_db /ibex/sw/rl9c/checkm2/1.0.2/rl9_conda3/database/CheckM2_database/uniref100.KO.1.dmnd \
    --gtdb_db /ibex/reference/KSL/gtdb-tk/release226/ \
    --refine_bins_dastool true \
    --postbinning_input both \
    --generate_bigmag_file true \
    -with-report reports/execution_report.html \
    -with-trace  reports/trace.tsv \
    -with-timeline reports/timeline.html
EOF

mkdir -p reports
sbatch run_nfcore_mag.sh
Always use -resume Adding -resume costs nothing on the first run (there is nothing cached yet) and saves hours on every re-run after a partial failure. Make it a habit to always include it.

After submitting, monitor the driver job and watch the pipeline progress:

# Check that the driver job is running
squeue -u $USER

# Tail the SLURM log for the Nextflow driver
tail -f nfcore_mag_*.log

# Check all child SLURM jobs submitted by Nextflow
squeue -u $USER --format="%.18i %.9P %.30j %.8u %.2t %.10M %.6D %R"

work/ — Process Sandboxes

Every process execution gets an isolated subdirectory under work/, named by a two-level hash (e.g. work/3a/f82c19.../). This is where Nextflow stages inputs, runs the tool, and captures outputs before publishing them to results/.

work/
└── 3a/
    └── f82c19d4b7e3a8c10f5b69e2d4f12345/
        ├── .command.sh       ← The exact shell command that was executed
        ├── .command.run      ← The SLURM wrapper script submitted to the scheduler
        ├── .command.log      ← Combined stdout + stderr from the tool
        ├── .command.out      ← stdout only
        ├── .command.err      ← stderr only
        ├── .command.env      ← Environment variables at execution time
        ├── .exitcode         ← Exit code (0 = success, non-zero = failure)
        ├── sample.fastq.gz -> /path/to/input  ← Symlinked inputs
        └── output_file.tsv   ← Actual output written by the tool
Inputs are always symlinks Nextflow never copies input files into the work directory — it creates symbolic links. This keeps disk usage low. If you delete the original input files before the pipeline finishes, you will break active symlinks and cause failures.

To find the work directory for a specific process, search the Nextflow log:

# Find work dir for a process by name
grep -i "GTDBTK" .nextflow.log | grep "work/"

# Or list all work dirs with their process names from the trace file
awk -F'\t' 'NR>1 {print $8, $1}' reports/trace.tsv | sort | head -40

Navigating Log Files

The .nextflow.log File

Nextflow writes a structured log to .nextflow.log in the launch directory, and rotates previous runs to .nextflow.log.1, .nextflow.log.2, etc. (up to five). This is the first place to look after any failure.

# Tail the live log while the pipeline is running
tail -f .nextflow.log

# Search for errors across the current log
grep -i "error\|FAILED\|exception" .nextflow.log

# Find which work dir a failed task used
grep -i "FAILED" .nextflow.log | grep -oP 'work/[a-f0-9]+/[a-f0-9]+'

Key patterns to look for in .nextflow.log:

PatternMeaning
Task FAILEDA process exited with a non-zero code — check .exitcode and .command.err in its work dir
Process > ... submittedA child SLURM job has been submitted for this process
cachedThis process was skipped because its output is already in cache (from a -resume run)
Workflow completedAll processes finished — look for ERROR just before this to see how many tasks failed
WARNNon-fatal issue (e.g. a deprecated parameter) that did not stop the run

Per-Process Log Files

Once you have the work directory path for a failed process, inspect its files directly:

# Navigate to the failed process work dir
cd work/3a/f82c19d4b7e3a8c10f5b69e2d4f12345

# What command was run?
cat .command.sh

# What did the tool print?
cat .command.log

# stderr only (most useful for errors)
cat .command.err

# What was the exit code?
cat .exitcode

# What environment variables were set?
grep "PATH\|SINGULARITY\|NXF" .command.env

SLURM Batch Driver Log

The SLURM output file for the Nextflow driver job (nfcore_mag_JOBID.log) contains the Nextflow run summary printed at the end of a successful or failed run:

N E X T F L O W  ~  version 24.10.2

Launching `https://github.com/nf-core/mag` [jolly_euler] DSL2 - revision: a1b2c3d4e5

executor >  slurm (342)
[48/9f3c21] process > NFCORE_MAG:MAG:FASTQC_RAW (mock_control)    [100%] 2 of 2 ✔
[3a/f82c19] process > NFCORE_MAG:MAG:FASTP (mock_control)         [100%] 2 of 2 ✔
[c7/2d89ab] process > NFCORE_MAG:MAG:MEGAHIT (group1)             [100%] 1 of 1 ✔
[d2/1e3f7c] process > NFCORE_MAG:MAG:GTDBTK_CLASSIFYWF (bins)    [ 50%] 1 of 2, failed: 1
...
ERROR ~ Error executing process › 'NFCORE_MAG:MAG:GTDBTK_CLASSIFYWF (bins)'

Caused by:
  Process `GTDBTK_CLASSIFYWF` terminated with an error exit status (137)

Work dir:
  /ibex/user/jdoe/metagenomics_course/nfcore_mag/work/d2/1e3f7c...
Exit code 137 means OOM-kill Linux exit code 137 = 128 + 9 (SIGKILL). The SLURM scheduler killed the process because it exceeded its memory allocation. Increase the memory for that process in your custom config (see Customising Config).

Troubleshooting Common Errors

Out-of-Memory (Exit 137)

# 1. Confirm it is an OOM kill
cat work/d2/1e3f7c.../.exitcode        # prints: 137
cat work/d2/1e3f7c.../.command.err     # often empty — kernel killed the process

# 2. Check how much memory the process requested vs used
grep "GTDBTK" reports/trace.tsv | awk -F'\t' '{print $5, $14, $15}'
# columns: name, peak_rss, peak_vmem

# 3. Fix: add a withName block in a custom config to increase memory
#    (see the "Customising Config" section below)

Walltime Exceeded (Exit 140 / DUE_TO_TIME_LIMIT)

# Confirm timeout
grep "TIME_LIMIT\|DUE_TO_TIME" .nextflow.log

cat work/xx/yyyyyy.../.command.err
# SLURM: Job XXXXXXX (process: NFCORE_MAG:MAG:MEGAHIT) timed out

# Fix: override time in a custom config
# process {
#   withName: 'MEGAHIT' { time = '12.h' }
# }

Missing Database File

# Symptom in .command.err:
Error: Database file not found: /path/to/checkm2_db/uniref100.KO.1.dmnd

# Fix: verify the path exists and is readable
ls -lh /ibex/sw/rl9c/checkm2/1.0.2/rl9_conda3/database/CheckM2_database/uniref100.KO.1.dmnd

# Then correct the --checkm2_db parameter and re-run with -resume

Singularity Container Pull Failure

# Symptom in .nextflow.log:
ERROR: Unable to pull container: 'docker://quay.io/biocontainers/...'
FATAL: Unable to handle docker://...

# This is usually a network issue on compute nodes.
# Check that the compute node has internet access via the proxy:
grep "SINGULARITY_PULLDIR\|http_proxy\|NXF_SINGULARITY" .command.env

# The kaust profile caches all containers to a shared directory.
# If a container is missing from cache, you may need to pre-pull:
module load singularity
singularity pull docker://quay.io/biocontainers/gtdbtk:2.4.0--pyhdfd78af_1

Resuming After a Failure

# 1. Fix the underlying issue (increase memory in custom config, fix path, etc.)
# 2. Re-submit the same SLURM script — -resume is already in the run command
sbatch run_nfcore_mag.sh

# Nextflow will skip all completed tasks and only retry the failed ones.
# Watch the log to confirm:
grep "cached\|submitted\|FAILED" .nextflow.log | tail -30
Do not delete the work/ directory between runs The work/ directory is Nextflow's cache. Deleting it forces every process to re-run from scratch even with -resume. Only clean it up when you are certain the run is complete and correct.

Execution Reports

Nextflow can generate three complementary reports that help you understand resource usage, bottlenecks, and the overall run timeline. Pass these flags to nextflow run:

FlagOutputUse
-with-report <file.html>Interactive HTML execution reportPer-task CPU, memory, wall time, I/O; spot OOM and timeout patterns
-with-trace <file.tsv>Tab-delimited trace fileMachine-readable resource usage; parse with awk or pandas
-with-timeline <file.html>Interactive Gantt chartVisualise which processes ran in parallel and where the pipeline stalled
-with-dag <file.html>Workflow DAG diagramUnderstand process dependencies at a glance

The nf-core/mag pipeline also automatically saves copies of these reports to results/pipeline_info/ after every run, even without the explicit flags.

Reading the HTML Execution Report

Open reports/execution_report.html in a browser. The report has three tabs:

  • Summary — total wall clock time, number of tasks, CPUs and memory requested vs used.
  • Resources — scatter plots of CPU % and memory usage per task. Tasks plotted far left of the requested line are over-provisioned; tasks touching the upper limit hit the ceiling.
  • Tasks — a sortable table of every task with its work directory path, peak RSS, and exit status. Click a row to jump to that work directory.

Parsing the Trace File

# Show the header
head -1 reports/trace.tsv | tr '\t' '\n' | cat -n

# Find the 10 most memory-hungry tasks
awk -F'\t' 'NR>1 {print $14, $1}' reports/trace.tsv \
  | sort -rh | head -10

# Find all failed tasks and their work dirs
awk -F'\t' '$7 != "COMPLETED" {print $7, $1, $8}' reports/trace.tsv

The KAUST Institutional Profile

The -profile kaust flag loads the KAUST institutional configuration from the nf-core/configs repository. It sets cluster-wide defaults so you do not have to specify them yourself every time:

// Excerpt from conf/kaust.config (nf-core/configs GitHub repo)
params {
    config_profile_description = 'KAUST Ibex HPC cluster profile'
    config_profile_contact     = 'KAUST Bioinformatics Core'
    max_memory                 = 1.TB
    max_cpus                   = 128
    max_time                   = 720.h
}

process {
    executor       = 'slurm'
    queue          = 'batch'
    scratch        = '/ibex/scratch/$USER'
    clusterOptions = '--mail-type=FAIL'
}

singularity {
    enabled    = true
    autoMounts = true
    cacheDir   = '/ibex/reference/KSL/singularity_cache'
}
Singularity cache on Ibex The kaust profile points Singularity to a shared container cache at /ibex/reference/KSL/singularity_cache. Containers already pulled by another user are reused automatically, so the first run is slower than subsequent ones.

Custom Config Files

Anything in the institutional profile can be overridden with a project-level custom.config file passed via the -c flag. This is the correct way to increase resource allocations, change queue names, or tweak tool arguments — without modifying the pipeline code itself.

# Run the pipeline with a custom config
nextflow run nf-core/mag \
    -r 5.4.2 \
    -profile kaust \
    -c custom.config \
    --input samplesheet.csv \
    --outdir results/

You can also place a nextflow.config in the launch directory — Nextflow picks it up automatically without the -c flag. This is handy for project-wide settings.

Changing Process Resources

Use the process { withName: '...' { ... } } block in your custom.config to override CPU, memory, and walltime for individual processes. Process names follow the pattern NFCORE_MAG:MAG:TOOLNAME; you can match by the short name or the full qualified name.

// custom.config — override resources for specific processes

process {

    // Increase memory for GTDB-Tk (needs ~200 GB for release 226)
    withName: 'GTDBTK_CLASSIFYWF' {
        cpus   = 32
        memory = 250.GB
        time   = '24.h'
    }

    // Give MEGAHIT more time for large co-assemblies
    withName: 'MEGAHIT' {
        cpus   = 48
        memory = 180.GB
        time   = '12.h'
    }

    // CheckM2 is I/O heavy; more CPUs help database lookup
    withName: 'CHECKM2_PREDICT' {
        cpus   = 16
        memory = 64.GB
        time   = '4.h'
    }

}
Process name matching Use a regular expression inside withName to target multiple processes at once — for example 'CONCOCT_.*' matches all CONCOCT sub-processes. Check the exact process names in the execution report or by running nextflow run nf-core/mag -r 5.4.2 --help 2>&1 | grep -i "process".

Dynamic Resource Scaling

Nextflow supports dynamic resource allocation using closures — expressions evaluated at run time. This lets you automatically escalate resources on retry:

// custom.config — dynamic resources with retry

process {

    // Double memory on each retry attempt; cap at 512 GB
    withName: 'GTDBTK_CLASSIFYWF' {
        errorStrategy = { task.exitStatus in [137, 140] ? 'retry' : 'finish' }
        maxRetries    = 2
        memory        = { min(128.GB * task.attempt, 512.GB) }
        time          = { 12.h * task.attempt }
    }

}

Changing Tool-Specific Parameters

Each nf-core process exposes an ext.args directive that is appended verbatim to the tool command line. This is the correct mechanism for passing flags that nf-core/mag does not expose as top-level pipeline parameters.

// custom.config — tool-specific arguments via ext.args

process {

    // Tune MEGAHIT k-mer sizes and minimum contig length
    withName: 'MEGAHIT' {
        ext.args = '--k-min 27 --k-max 127 --k-step 10 --min-contig-len 1000'
    }

    // Lower MetaBAT2 minimum contig size (default: 2500)
    withName: 'METABAT2_METABAT2' {
        ext.args = '--minContig 1500 --seed 1'
    }

    // Run SemiBin2 in single-sample mode (default: multi)
    withName: 'SEMIBIN_SEMIBIN2' {
        ext.args = '--self-supervised'
    }

    // Pass extra Prokka flags for compliance mode
    withName: 'PROKKA' {
        ext.args = '--compliant --centre KAUST --mincontiglen 500'
    }

    // Increase DAS Tool score threshold (default: 0.5; range 0–1)
    withName: 'DASTOOL_DASTOOL' {
        ext.args = '--score_threshold 0.6 --megabin_penalty 0.02'
    }

}
Check the module source before using ext.args Not all tool wrappers honour ext.args in the same position — some place it before inputs, some after. Browse the process .nf file in the nf-core/mag GitHub repository (modules/nf-core/<toolname>/main.nf) to confirm where $options.args or $args appears in the command string.

Full custom.config Example

A single custom.config can combine resource overrides, retry logic, and ext.args in one file:

// custom.config — full example for KAUST Ibex course run

params {
    // Raise global caps beyond the kaust profile defaults if needed
    max_memory = 512.GB
    max_cpus   = 64
    max_time   = '168.h'
}

process {

    // ── Assembly ─────────────────────────────────────────────
    withName: 'MEGAHIT' {
        cpus     = 48
        memory   = 180.GB
        time     = '12.h'
        ext.args = '--k-min 27 --k-max 127 --k-step 10'
    }

    // ── Binning ───────────────────────────────────────────────
    withName: 'METABAT2_METABAT2' {
        cpus     = 16
        memory   = 64.GB
        ext.args = '--minContig 1500'
    }

    withName: 'SEMIBIN_SEMIBIN2' {
        cpus   = 16
        memory = 128.GB
        time   = '8.h'
    }

    withName: 'CONCOCT_CUTUPFASTA|CONCOCT_CONCOCT|CONCOCT_MERGECUTUPCLUSTERING' {
        cpus   = 8
        memory = 32.GB
    }

    // ── Bin refinement ────────────────────────────────────────
    withName: 'DASTOOL_DASTOOL' {
        cpus     = 16
        memory   = 64.GB
        ext.args = '--score_threshold 0.6'
    }

    // ── Bin QC ────────────────────────────────────────────────
    withName: 'CHECKM2_PREDICT' {
        cpus   = 16
        memory = 64.GB
        time   = '6.h'
    }

    // ── Taxonomy ──────────────────────────────────────────────
    withName: 'GTDBTK_CLASSIFYWF' {
        cpus          = 32
        memory        = 250.GB
        time          = '24.h'
        errorStrategy = { task.exitStatus == 137 ? 'retry' : 'finish' }
        maxRetries    = 2
    }

    // ── Annotation ────────────────────────────────────────────
    withName: 'PROKKA' {
        cpus     = 8
        memory   = 32.GB
        ext.args = '--compliant --centre KAUST'
    }

}

DAY 1, 09:15 – 10:15

Linux Command Line & Ibex Cluster

Get connected to the Ibex HPC cluster, explore the course data, and learn the essential Linux commands and SLURM job submission patterns you will use throughout the week.

Learning Objectives

  • Connect to Ibex via SSH and verify your environment
  • Understand Ibex storage tiers and where to put course files
  • Inspect FASTQ and TSV files using standard Linux tools
  • Activate and manage conda environments for course tools
  • Submit, monitor, and cancel SLURM jobs
  • Download public sequencing data from NCBI SRA using prefetch and fasterq-dump
  • Download reference genomes using NCBI Datasets CLI
  • Use screen to keep sessions alive during long runs

Step 1: Connect to Ibex

Ibex has three login nodes for different purposes. For this course use ilogin (CPU jobs):

Step 3: Set Up Your Working Directory

Use /ibex/user/$USER for all course work: it has 1.5 TB, high-performance WekaIO storage, and persists for the lifetime of your account.

# Create your personal working directory
mkdir -p /ibex/user/$USER/metagenomics_course
cd /ibex/user/$USER/metagenomics_course

# Create a symlink to the course data
ln -s $COURSE_DIR data

# Add a WORKDIR variable to your shell profile
echo 'export WORKDIR=/ibex/user/$USER/metagenomics_course' >> ~/.bashrc
source ~/.bashrc

# Verify
echo $WORKDIR
ls $WORKDIR/data

Step 4: Essential Commands for This Course

Practice these commands on the course FASTQ files: you will use them in every module.

Viewing and inspecting files

# Peek at a gzip-compressed FASTQ (4 lines = 1 read)
zcat data/reads/sample1_R1.fastq.gz | head -8

# Count the number of reads  (lines ÷ 4)
zcat data/reads/sample1_R1.fastq.gz | wc -l

# Scroll through a file interactively  (q to quit)
zcat data/reads/sample1_R1.fastq.gz | less

# Show file sizes
ls -lh data/reads/

Searching and filtering

# Count read headers (lines starting with @)
zcat data/reads/sample1_R1.fastq.gz | grep -c "^@"

# Show only the first column of a TSV file
cut -f1 data/metadata/samples.tsv | head -10

# Search a text file for a pattern
grep "sample1" data/metadata/samples.tsv

Working with directories

# Create a nested directory structure
mkdir -p results/01-qc/fastqc

# Show a tree-like directory listing
ls -R results/

# Move between directories
cd results/01-qc
pwd
cd $WORKDIR         # back to working directory

Step 6: SLURM Job Submission

All computational work must run on compute nodes via SLURM, never directly on the login node. Submit CPU jobs from ilogin. The default partition is batch.

Interactive session

Use srun for interactive hands-on steps. This gives you a shell on a compute node:

srun --partition=batch \
     --time=02:00:00 \
     --mem=16G \
     --cpus-per-task=8 \
     --pty bash

Batch job

Use sbatch for long-running steps (assembly, GTDB-Tk) that you do not need to watch interactively:

cat > my_job.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=my_job
#SBATCH --partition=batch
#SBATCH --time=04:00:00
#SBATCH --mem=32G
#SBATCH --cpus-per-task=16
#SBATCH --output=my_job_%j.out
#SBATCH --error=my_job_%j.err
#SBATCH --mail-type=END,FAIL
#SBATCH --mail-user=<your@email.com>

source ~/.bashrc
conda activate metagenomics-assembly

# your command here
EOF

sbatch my_job.sh
Targeting specific node types Use --constraint to request specific CPU architectures. For high-memory steps (SPAdes, GTDB-Tk) request a large-memory node: --constraint="largemem". Run sinfo to see all available node types and current availability.

Monitoring and cancelling jobs

# List your running and pending jobs
squeue -u $USER

# Show detailed info about a job
scontrol show job <job_id>

# Cancel a job
scancel <job_id>

# Show all available nodes and partitions
sinfo

Fair-share scheduling Ibex uses a fair-share allocation system. Heavy recent usage lowers your scheduling priority. For the course, keep job requests reasonable: use only the CPUs and memory you actually need.

Reference Databases on Ibex

Many bioinformatics databases (GTDB-Tk, CheckM2, BLAST, and others) are pre-installed on Ibex and available to all users: you do not need to download them yourself. Access requires a one-time request that is approved automatically.

  2. Search for the database you need and click Request Access

Step 7: Downloading Public Data from NCBI

Two complementary tools cover the most common data-download scenarios: SRA Toolkit for raw sequencing reads (FASTQ) and NCBI Datasets CLI for reference genomes and assemblies.

Part A: SRA Toolkit (reads)

The NCBI Sequence Read Archive (SRA) is the world's largest repository of high-throughput sequencing data. Each experiment is identified by a run accession starting with SRR. SRA Toolkit provides two key commands:

CommandWhat it does
prefetchDownloads the compressed SRA container file to local cache. Resumable if interrupted: always use this step first for large files.
fasterq-dumpConverts the SRA container to FASTQ. --split-files writes R1/R2 as separate files; -p shows a progress bar; -e sets thread count.
vdb-dump --infoInspects an SRA file or accession before converting: shows read count, read length, library layout.
Why prefetch first? Running fasterq-dump directly on an accession streams data over the network and cannot resume. prefetch downloads a single compressed .sra file that can restart from where it stopped; fasterq-dump then converts locally, which is also faster.

Inspect an accession before downloading:

module load sratoolkit

# Inspect read count, layout, and length before committing to a download
vdb-dump --info SRR9291172
Module not found error If module load TOOL_XYZ fails, search in the available app stack using module avail KEYWORD e.g. module avail sra to find the correct name of the app.

Download and convert the course dataset (PRJNA521492: Pereira-Marques et al. 2019):

cd /ibex/user/$USER/metagenomics_course
mkdir -p raw_reads
cd raw_reads

# Print prefetch help
prefetch --help

# Step 1: Download the SRA container (resumable)
prefetch --progress SRR9291172 SRR9291174

# Print fasterq-dump help
fasterq-dump --help

# Step 2: Convert to FASTQ (split paired reads, show progress, 16 threads)
fasterq-dump SRR9291172 \
    --split-files \
    --outfile mock_control \
    -p -v -e 16

fasterq-dump SRR9291174 \
    --split-files \
    --outfile mock_ss90 \
    -p -v -e 16

# Step 3: Compress (most tools accept .fastq.gz; ~4× size reduction)
gzip mock_control_1.fastq mock_control_2.fastq &
gzip mock_ss90_1.fastq mock_ss90_2.fastq &

wait;

# Step 4: Clean up SRA containers and temp files
rm -rf SRR9291172/ SRR9291174/ fasterq.tmp.*

# Verify
ls -lh *.fastq.gz
# or
tree -h ./
Expected files after download mock_control_1.fastq.gz, mock_control_2.fastq.gz: 0 % host (mock community only)
mock_ss90_1.fastq.gz, mock_ss90_2.fastq.gz: 90 % C57BL/6 mouse host DNA
Course shortcut The raw data for this course are already pre-staged at /biocorelab/BIX/resources/training/XXX. You only need the steps above if you want to practise the download workflow or need data not included in the course data.

Part B: NCBI Datasets CLI (reference genomes)

NCBI Datasets is the modern replacement for Entrez Direct for downloading genomes and assemblies. Accessions starting with GCA_ (GenBank) or GCF_ (RefSeq) identify genome assemblies.

module load ncbi-datasets-cli

# Inspect assembly metadata before downloading
datasets summary genome accession GCA_000001635.9 | python3 -m json.tool | head -40

Download the C57BL/6 mouse reference genome (GRCm39) used in Module 1 for host removal:

mkdir -p /ibex/user/$USER/metagenomics_course/references
cd /ibex/user/$USER/metagenomics_course/references

# Download genome FASTA only
datasets download genome accession GCA_000001635.9 \
    --include genome \
    --filename GRCm39.zip

# Unzip and move the FASTA
unzip GRCm39.zip
# Inspect content
tree -h ncbi_dataset
# move reference fasta file
mv ncbi_dataset/data/GCA_000001635.9/*.fna GRCm39_GCA_000001635.9.fa
# Clean-up
rm -rf ncbi_dataset GRCm39.zip
Dehydrated downloads for large genomes For very large genomes or many assemblies add --dehydrated to download only metadata first, then datasets rehydrate --directory . to fetch the sequences. This is useful on slow connections or when downloading hundreds of assemblies.
Course shortcut The GRCm39 reference genome is already pre-staged at /biocorelab/BIX/resources/genomes/Ensembl-release-113/GRCm39/GRCm39.fa. You only need the steps above if you want to practise the download workflow or need a genome not included in the course data.

Part C: Entrez Direct (metadata and accession lists)

Entrez Direct (esearch / efetch) is useful for querying NCBI databases programmatically: especially for fetching run metadata or generating accession lists for a BioProject before downloading with SRA Toolkit.

module load entrez-direct

# Fetch all SRA run accessions for a BioProject
esearch -db sra -query PRJNA521492 | efetch -format runinfo > PRJNA521492_runinfo.csv

# Preview the columns
head -2 PRJNA521492_runinfo.csv | cut -d',' -f1-6
Known issue on Ibex: SSL error On some Ibex nodes esearch fails with OpenSSL error:0A00006E: bad extension. This is a TLS compatibility issue between the cluster's OpenSSL build and NCBI's servers. If you hit this, use the SRA Run Selector web interface instead: go to ncbi.nlm.nih.gov/Traces/study/, search for your BioProject accession, select the runs you want, and click Accession List to download a plain-text SRR_Acc_List.txt file. You can then feed it directly to prefetch --option-file SRR_Acc_List.txt.

For bulk downloads of the actual reads, extract the Run column and pass accessions to prefetch; for metadata and genome downloads, prefer NCBI Datasets as shown in Part B above.

Step 8: Keep Sessions Alive with Screen

If your SSH connection drops, an interactive srun session is lost. Use screen to protect long sessions.

# Start a named screen session on the login node
screen -S metagenomics

# Inside screen: launch your srun / work as normal
# Detach from screen (does NOT kill it):  Ctrl+A  then  D

# List all screen sessions
screen -ls

# Re-attach to your session after reconnecting
screen -r metagenomics
Tip: screen inside srun Start screen on the login node first, then run srun inside it. This way, if your laptop disconnects, the interactive compute session keeps running and you can re-attach from any login node.
